# Skill Sediment — 技能沉淀流程

把当前对话中反复出现的修正、验证过的技巧、以及发现的坏模式，写进对应的 skill 文件。

**参数**：`$ARGUMENTS`（skill 名称，如 `nonfiction-refine`，或完整路径）

---

## 为什么需要这个流程

Claude 默认行为 ≠ skill 期望行为。每次重新调用 skill 时，如果修正只停留在对话里，下次还会犯同样的错误。

**沉淀的本质**：把"用户纠正了我3次的事"变成"skill 的明确规则"。

---

## 第一步：定位 skill 文件

```bash
# 在常见 skill 目录下查找
find ~/.claude/skills/ -name "SKILL.md" | xargs grep -l "<skill名>"
find /Users/changxing/project/OnCourse/platform/landpages/LayeredVisions/.claude/skills/ -name "SKILL.md"
find /Users/changxing/project/OnCourse/platform/one-person-company-skills/ -name "SKILL.md"
```

读取 skill 目录下的所有文件，了解当前结构。

---

## 第二步：提炼本次对话的修正

从当前对话上下文中，按优先级找出以下三类内容：

### 类型 A：反复纠正（最高优先级）
用户对同一问题纠正了 ≥2 次 → **必须写成明确禁止规则**

识别模式：
- "不是这样的"、"又犯了"、"还是没改"
- 用户改了 Claude 的输出，Claude 下次又用旧方式

写法：`**禁止**：[具体行为]。用 [正确方式] 代替。`

### 类型 B：验证有效的技巧（重要）
用户确认了某个非显而易见的方法有效 → **加入方法论或检查清单**

识别模式：
- "对，就是这样"、"这个好"、"保留这个"
- 用户接受了某个不寻常的选择没有反对

写法：把技巧加进相关的检查项或步骤说明。

### 类型 C：发现的坏模式（常见）
在实际执行中发现的典型错误，之前检查清单没有覆盖 → **加进诊断检查项**

写法：在检查清单中加 `- [ ] [新检查项]`，注明为什么这个常被漏掉。

---

## 第三步：确认要写入的位置

根据内容类型，选择写入位置：

| 内容类型 | 写入位置 |
|---------|---------|
| 流程规则、禁止行为、输出格式 | `SKILL.md` 主文件 |
| 好写法 vs 坏写法的具体例子 | `writing-examples.md`（或对应示例文件） |
| 调研方法、信息来源策略 | `research-guide.md`（或对应指南文件） |
| 人物/风格参考 | `writer-souls.md`（或对应参考文件） |

如果 skill 只有单文件，全部写入主文件。

---

## 第四步：写入修改

**写入原则**：

1. **位置要精准**：修正要放在最能被读到的地方。如果流程第一步就会犯错，规则要放在第一步，不是附录。

2. **不加"注意"或"记住"**：直接写规则本身。"**修复 = 直接重写，不是建议。**" 比 "请记住修复要直接重写" 有力得多。

3. **坏例子比规则更有效**：如果有具体的坏模式，写成 `坏：[具体例子]` / `好：[改后版本]` 的对照格式。

4. **不要堆砌**：每次沉淀只加真正通过实践验证的内容，不加"也许有用"的猜测。

5. **更新而不是追加**：如果已有相关规则，更新它，不要加一条类似的重复规则。

用 Edit 工具直接修改文件，不要只列出修改建议。

---

## 第五步：验证写入质量

写完后，重读被修改的段落，自问：

- **这个规则能防止上次那个错误吗？** 如果规则太模糊，继续具体化。
- **规则是可执行的吗？** "更有深度"不是规则。"最深的洞见必须展开为至少3步逻辑链条"是规则。
- **位置是对的吗？** 规则在流程中遇到时才有效，放在结尾的规则往往被忽视。

---

## 输出

完成后，简要报告：

```
## 沉淀结果

### 修改了哪些文件
- [文件路径]：[一行说明改了什么]

### 沉淀了哪些内容
- 类型A（禁止规则）：[具体规则]
- 类型B（验证技巧）：[具体技巧]
- 类型C（新检查项）：[具体检查项]

### 下次调用该 skill 时，哪些错误不会再犯
- [错误1]
- [错误2]
```
